## Основные изменения в API

### Глобальный рефакторинг путей
- **Старые пути**: `/rvc/{rvcId}/egais/*`
- **Новые пути**: `/rvc/{rvcId}/fiscaldb/*`

### Методы для работы с марками ЕГАИС

#### Добавление марок
- `POST /rvc/{rvcId}/fiscaldb/marks` - добавить одну марку
- `POST /rvc/{rvcId}/fiscaldb/marks/batch` - пакетное добавление марок

#### Получение марок
- `GET /rvc/{rvcId}/fiscaldb/marks/{code}` - получить марку по коду
- `GET /rvc/{rvcId}/fiscaldb/marks/byobjnum/{objnum}` - получить марки для позиции меню
- `GET /rvc/{rvcId}/fiscaldb/marks/sold` - выгрузка марок из проданных чеков

#### Удаление марок
- `DELETE /rvc/{rvcId}/fiscaldb/marks/{code}` - удалить марку

### Методы для обновления позиций меню
- `PUT /rvc/{rvcId}/fiscaldb/menuitems/egais` - обновить ЕГАИС-данные позиции меню
- `PUT /rvc/{rvcId}/fiscaldb/menuitems/egais/batch` - пакетное обновление ЕГАИС-данных

### ❌ Удалённые методы (не используются!)
Все методы работы со штрихкодами через `/fiscaldb/barcodes/*` больше не используются.

## Структуры данных

### FiscalDbAddMarkDto (добавление марки)

```json
{
  "objnum": 1005,
  "code": "187320871162320623001PBSYDRJXFRMZ6WRPHHPGJSXSCYBUMY3TUXIHCVVB2LM73XU4Y5VNRCWBWSF3XYC533W5RBIMSA",
  "store": "Demo",
  "name": "Водка русская тестовая 0.7л Без EAN ДЕМО",
  "ean": " ",
  "tare": "БУТЫЛКА",
  "volume": 0.7,
  "remainder": 0.7,
  "state": "",
  "keeptime": 0
}
```

**Обязательные поля:**
- `objnum` - ID позиции меню в Simphony
- `code` - код акцизной марки ЕГАИС

**Особенности для бутылок:**
- `ean` - всегда пробел `" "`
- `tare` - всегда `"БУТЫЛКА"`
- `state` - пустая строка
- `keeptime` - всегда `0`

### UpdateMenuItemFiscalDbDto (обновление позиции меню)

```json
{
  "objnum": 1006,
  "volume": 0.7,
  "unit": 41,
  "marktype": 1,
  "groupid": 0,
  "cocktid": 0
}
```

**Поля:**
- `objnum` - ID позиции меню
- `volume` - объём в литрах/килограммах
- `unit` - единица измерения:
  - `0` - штуки (для блюд)
  - `10` - граммы
  - `11` - килограммы
  - `40` - миллилитры
  - `41` - литры
- `marktype` - тип марки:
  - `1` - бутылка
  - `2` - порция
  - `3` - коктейль
  - `4` - разливное пиво
  - `5` - кег

### FiscalDBSoldMark (проданная марка)

```json
{
  "id": 12345,
  "cknum": 191131,
  "ckpos": 1,
  "objnum": 1005,
  "code": "187320871...",
  "vol": 0.05,
  "date": "2026-03-17T15:30:00",
  "rvc": 160,
  "name": "Водка..."
}
```

**⭐ Ключевые поля для сопоставления:**
- **`cknum`** - номер чека для поиска документа "Чек ККМ" в 1С
- **`ckpos`** - номер позиции в чеке для поиска строки в ТЧ "Товары" (НомерСтроки = ckpos)
- **`vol`** - проданный объём (СПИСАНИЕ!), записывается в ТЧ "МаркиЕГАИС"
- **`code`** - код марки для записи в ТЧ "МаркиЕГАИС"

## Идентификаторы для API

### Проблема множественных идентификаторов

В API Simphony используются разные идентификаторы для разных операций:

1. **`hierstrucid`** - создание кнопок меню через `/menu-item`
2. **`rvc_id точки продаж`** - обновление ЕГАИС-атрибутов через `/fiscaldb/menuitems/egais`
3. **`parent_hierstrucid (rvc_id ветки)`** - выгрузка марок через `/fiscaldb/marks`

### Пример структуры

```json
{
  "hierstrucid": 142,
  "parent_hierstrucid": 58,
  "rvc_id": "160",
  "rvc_name": "Vysota1234"
}
```

### Использование в API-методах

| Метод | Идентификатор | Комментарий |
|-------|---------------|-------------|
| `POST /menu-item` | `hierstrucid` | Создание кнопки меню |
| `PUT /rvc/{rvcId}/fiscaldb/menuitems/egais` | `rvc_id точки` | Обновление ЕГАИС-атрибутов |
| `POST /rvc/{rvcId}/fiscaldb/marks` | `parent_hierstrucid` | Выгрузка марок (rvc_id ветки!) |

**Важно:** При выгрузке марок в `rvc_id` передаётся `parent_hierstrucid` (ID ветки), а в поле `store` - наименование точки продаж.

## Критичные моменты

### ⚠️ 1. Поле `vol` - это ПРОДАННЫЙ ОБЪЁМ (списание)!

```json
{
  "code": "187320...",
  "vol": 0.05  // ← Объём СПИСАНИЯ (проданный объём)
}
```

**Архитектура FiscalDB:**
- Таблица `store (markcodes)` - хранит марки и их остатки на складе
- Таблица `excisemarks` - хранит транзакции списания марок
- API `/marks/sold` - возвращает JOIN этих таблиц, где `vol` из `excisemarks`

**Формула:**
```bsl
ОбъёмПродано = vol  // Напрямую из API!
```

**Пример:**
- Была бутылка 0.7л на складе
- Продали порцию 50мл (0.05л)
- API вернёт `vol = 0.05` (проданный объём) ✅

### ⚠️ 2. Ключ сопоставления: cknum + ckpos

**КРИТИЧНО:** Сопоставление марок со строками чека происходит по **двум** полям:
- `cknum` - номер чека в Simphony
- `ckpos` - номер позиции (строки) в чеке

**Пример:**
```
Чек №191134:
├─ Позиция 1 (ckpos=1): Водка порция 50мл
├─ Позиция 2 (ckpos=2): Пиво разливное 0.5л
└─ Позиция 3 (ckpos=3): Водка порция 50мл

Марки:
├─ cknum=191134, ckpos=1 → привязана к первой позиции ✅
├─ cknum=191134, ckpos=2 → привязана ко второй позиции ✅
└─ cknum=191134, ckpos=3 → привязана к третьей позиции ✅
```

**Сценарий:** Продали 50мл водки из двух бутылок (30мл + 20мл)

```json
[
  {
    "cknum": 191134,
    "ckpos": 1,
    "code": "MARK1...",
    "vol": 0.03
  },
  {
    "cknum": 191134,
    "ckpos": 1,
    "code": "MARK2...",
    "vol": 0.02
  }
]
```

**Обработка:**
```bsl
// ОБЯЗАТЕЛЬНО группировать по cknum + ckpos!
КлючСопоставления = Формат(cknum, "ЧГ=0") + "_" + Формат(ckpos, "ЧГ=0");

// Для ключа "191134_1":
// 1. Найти документ Чек ККМ по cknum = 191134
// 2. Найти строку в ТЧ Товары где НомерСтроки = ckpos = 1
// 3. Добавить ДВЕ марки в ТЧ МаркиЕГАИС с НомерПозицииЧека = 1
// 4. Проверить: Σ(vol) = 0.03 + 0.02 = 0.05л = объём позиции ✅
```

### ⚠️ 3. НЕ нужен регистр остатков марок!

API уже возвращает проданный объём напрямую в поле `vol`.

**Структура в 1С:**
```
ТЧ: МаркиЕГАИС в документе "Чек ККМ"
- НомерПозицииЧека (Число 4)
- КодМарки (Строка 150)
- ОбъёмПродано (Число 15.3)  // Берём напрямую из vol!
```

### ⚠️ 4. Валидация целостности

```bsl
СуммаПоМаркам = Σ(МаркиЕГАИС.ОбъёмПродано)  // Сумма vol из API
ОжидаемыйОбъём = Товары.Количество × Номенклатура.ОбъемДАЛ

Если |СуммаПоМаркам - ОжидаемыйОбъём| > 0.001 Тогда
    // ❌ Ошибка! Объёмы не сходятся
КонецЕсли;
```

### ⚠️ 5. Типичные ошибки

#### ❌ Ошибка 1: Думать что vol - это остаток
```bsl
// НЕПРАВИЛЬНО:
НачальныйОстаток = ПолучитьОстатокМарки(КодМарки);
ОбъёмПродано = НачальныйОстаток - СтрокаМарки.vol;  // ❌

// ПРАВИЛЬНО:
ОбъёмПродано = СтрокаМарки.vol;  // ✅ Напрямую из API
```

#### ❌ Ошибка 2: Не использовать ckpos
```bsl
// НЕПРАВИЛЬНО:
Для Каждого СтрокаМарки Из ТаблицаМарок Цикл
    Чек = НайтиЧек(СтрокаМарки.cknum);
    Чек.МаркиЕГАИС.Добавить();  // К какой позиции? ❌
КонецЦикла;

// ПРАВИЛЬНО:
МаркиПоПозициям = СгруппироватьПо_cknum_ckpos(ТаблицаМарок);
Для Каждого Позиция Из МаркиПоПозициям Цикл
    Чек = НайтиЧек(Позиция.cknum);
    СтрокаТовара = Чек.Товары.Найти(Позиция.ckpos, "НомерСтроки");  // ✅
    // ...
КонецЦикла;
```

## Перезагрузка чеков

### Параметры запроса

API `/fiscaldb/marks/sold` поддерживает:
- `dateFrom` / `dateTo` - период загрузки
- `cknum` - конкретный номер чека

### Сценарии использования

#### 1. Массовая загрузка за период
```http
GET /rvc/160/fiscaldb/marks/sold?dateFrom=2026-03-17T00:00:00&dateTo=2026-03-17T23:59:59
```

**Когда:** Регулярная автоматическая загрузка (раз в час, в конце смены).

#### 2. Перезагрузка конкретного чека
```http
GET /rvc/160/fiscaldb/marks/sold?cknum=191131
```

**Когда:**
- Обнаружена ошибка в конкретном чеке
- Нужна отладка проблемы с марками
- После исправления данных в Simphony

#### 3. Комбинированный запрос
```http
GET /rvc/160/fiscaldb/marks/sold?cknum=191131&dateFrom=2026-03-17T00:00:00&dateTo=2026-03-17T23:59:59
```

**Когда:** Дополнительная фильтрация для безопасности.

### Использование в 1С

```bsl
// Массовая загрузка
ПараметрыВыборки.НомерЧека = 0;  // или Неопределено
ТаблицаМарок = ПолучитьПроданныеМаркиHTTP(ПараметрыВыборки);

// Перезагрузка конкретного чека
ПараметрыВыборки.НомерЧека = 191131;
ТаблицаМарок = ПолучитьПроданныеМаркиHTTP(ПараметрыВыборки);
```

## Алгоритм обработки марок (упрощённый)

```
1. Загрузить чеки за период (или конкретный чек)
   └→ GET /checks/check-details?dateFrom=...&dateTo=...
      или
   └→ GET /checks/check?cknum=...

2. Загрузить марки аналогично
   └→ GET /rvc/{id}/fiscaldb/marks/sold?dateFrom=...&dateTo=...
      или
   └→ GET /rvc/{id}/fiscaldb/marks/sold?cknum=...

3. Отсортировать по чеку/позиции/времени
   └→ ТаблицаМарок.Сортировать("cknum, ckpos, date")

4. Сгруппировать по позициям (cknum + ckpos)
   └→ Соответствие: "cknum_ckpos" → Массив марок

5. Для каждой марки:
   ├→ Взять проданный объём = vol (напрямую из API!)
   ├→ Привязать к позиции чека через ckpos
   └→ Записать в ТЧ МаркиЕГАИС

6. Валидация:
   └→ Σ(vol) для позиции = объём позиции чека
```

- Валидировать суммы для КАЖДОЙ позиции отдельно
- Логировать режим загрузки (массовый / конкретный чек)
- Обрабатывать ошибки для каждой точки продаж отдельно

### ❌ Не делать:

- Сопоставлять марки только по `cknum` (без `ckpos`)
- Создавать регистр остатков марок
- Валидировать весь чек целиком (вместо каждой позиции)
